# Evidence — live researcher → critic → writer run on coilmem

These artifacts are the durable output of one **live** team session (real LLMs, real
embeddings) that exercises coilmem's shared-memory wedge. They exist to answer one
question with proof rather than assertion: **does `shared` memory actually carry
context between agents, while `private` stays isolated?**

## Run parameters (what produced these)

| | |
|---|---|
| Workspace | `learning-ai` |
| Turns (topics) | 3 — ML foundations → hands-on projects → staying current |
| Roles | researcher → critic → writer (sequential, per turn) |
| Models | researcher=`openai:gpt-4o-mini`, critic=`anthropic:claude-haiku-4-5`, writer=`openai:gpt-4o` |
| Memory mode | `memory` (recall = top-k of `shared` + caller's own `private`), k=5 |
| Embeddings | `openai` text-embedding-3-small |
| Totals | 18 memories (9 shared / 9 private), 9 recalls, 9 LLM calls, ~20.7k tokens |
| Headline | **20 cross-agent shared retrievals, 0 foreign-private leaks** |

Generated by `evidence_run.py` (the run + raw dumps), `generate_trace.py` (`trace.md`),
and `generate_memories_md.py` (`memories_full.md`). Re-run any of them to reproduce.

## Two lenses — read this first

Every file is one of two views, and conflating them is the main interpretation trap:

- **Agent view** = what an agent actually received, *after* the scope rule filtered it.
  Source of truth: `recall_log.json` (and `report.md`). An agent only ever sees
  `shared` + its **own** `private`.
- **Operator view** = the raw table, *unfiltered*, what someone with DB access sees.
  Source of truth: `memories.json` / `memories_full.md`. Contains everyone's private
  rows — which no agent could retrieve.

The product guarantee is the *gap* between these two: the operator view holds 9 private
rows; the agent view never surfaced one across owners.

## Files, what each surfaces, and how to read it

### `trace.md` — START HERE — the causal proof
**Surfaces:** a single idea travelling researcher → critic → writer through shared memory.
**How to read:** Section 1 is a ledger of 7 ML terms marked ✗/✓/✓ across the three agents.
The pattern `✗ ✓ ✓` means a term is **absent from the researcher** but **present in both the
critic and the writer** — so the writer's only possible source was the critic's `shared`
memory. Section 2 shows the recall log line proving the writer retrieved the critic before
writing. Sections 3–4 give the three quoted texts and the inference chain. If you read one
file, read this — it's the "shared memory helped, here's exactly how" exhibit.

### `report.md` — per-turn recall summary (human-readable)
**Surfaces:** for each of the 9 recalls, who recalled, and a tagged list of what they got.
**How to read:** each hit is `[author/SCOPE score=X] text`. `CROSS-SHARED` = a `shared`
memory written by a *different* agent (the feature working). `score` is cosine relevance
(higher = more relevant; that's why retrieval is ranked, not a dump). The per-recall header
counts `cross-agent SHARED | own | foreign PRIVATE leaked`. Watch two things: (1) the
counts climb 0→1→2 across researcher→critic→writer in Turn 1 (the pipeline filling up via
memory); (2) `foreign PRIVATE leaked` is **0 everywhere** — and any `PRIVATE` line that does
appear has `author == caller` (an agent seeing its *own* private, which is allowed).

### `recall_log.json` — raw recall events (machine-readable)
**Surfaces:** the same 9 recalls as `report.md`, untruncated, for scripting/verification.
**How to read:** a list of `{turn, caller, topic, recalled:[{author, scope, score, text}]}`.
This is the **agent view** ground truth — exactly what entered each agent's prompt. To
re-verify isolation yourself: assert no element has `scope=="private" and author!=caller`.
To re-verify sharing: count elements with `scope=="shared" and author!=caller` (= 20).

### `memories_full.md` — every memory in full (operator view, readable)
**Surfaces:** all 18 stored memories, complete text, grouped Turn → role → scope.
**How to read:** the 9 `shared` entries are the real model outputs (2–5 KB each — the actual
findings, critiques, sections). The 9 `private` entries are one-line placeholders
(`"<role> private scratch on <topic>"`) the team writes so there is something private to
isolate. This is the **operator view**: it contains rows no agent could retrieve. Cross-check
against `recall_log.json` to see that these private rows never reached another agent.

### `memories.json` — raw memory rows (machine-readable)
**Surfaces:** the same 18 memories as `memories_full.md`, as structured records.
**How to read:** `{agent_id, scope, text, metadata, created}`. Two tells worth noting:
`shared` rows carry `metadata={"turn","topic"}` (provenance), `private` rows have
`metadata=null`; and `created` timestamps are strictly ordered researcher→critic→writer
within each turn — the pipeline executing for real. This is the operator-view ground truth.

### `sections.json` — full assembled outputs per turn
**Surfaces:** each turn's three role outputs as delivered (the "product" of the run).
**How to read:** a list of `{turn, topic, outputs:{researcher, critic, writer}}`. Use this to
read the writer's final section in context and confirm it incorporates upstream content —
the readable companion to the `trace.md` argument.

## Suggested reading order

1. `trace.md` — see the mechanism prove itself on one idea.
2. `report.md` — see it generalize across all 9 recalls.
3. `memories_full.md` — read the actual content that moved.
4. `recall_log.json` / `memories.json` — re-verify the claims by script.
5. `sections.json` — read the finished outputs.

## Reproduce

From the repo root, with provider keys in `.env`:
```
python tools/evidence_run.py          # writes memories.json, recall_log.json, sections.json, report.md
python tools/generate_trace.py        # writes trace.md
python tools/generate_memories_md.py  # writes memories_full.md
```
(The harness defaults `EMBED_PROVIDER` to `openai`; the `.env` loader strips inline `#` comments.)
